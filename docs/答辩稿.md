## 一、开场（约1分钟）

各位老师好，我是 XXX，今天我答辩的项目是 **University Helper —— 校园在线学习辅助平台**。

这个项目的出发点很简单：当前高校广泛使用超星学习通、智慧树等在线学习平台，学生在日常使用中面临签到方式多样（定位、拍照、二维码、手势等）、课程视频繁多、章节测验重复等痛点。University Helper 旨在提供一个统一的、自动化的辅助工具，帮助学生更高效地管理在线学习任务。

## 二、技术架构（约3分钟）

本项目采用前后端分离的全栈架构，主要技术选型如下：

**后端：**
- 语言与框架：Python 3.10 + FastAPI
- 数据校验：Pydantic v2
- 数据库驱动：psycopg2（原生 SQL，未使用 ORM，追求更精细的查询控制）
- 认证方案：JWT（PyJWT）+ bcrypt 密码哈希
- 网络请求：httpx + requests，用于与第三方平台交互
- 页面解析：BeautifulSoup4 + lxml
- 加密模块：pycryptodome + pyaes，兼容超星/智慧树的加密协议
- 反爬处理：fonttools，解码超星自定义字体反爬机制

**前端：**
- 框架：React 18 + Vite 5
- 路由：React Router v6
- 样式：Tailwind CSS
- 状态管理：Zustand
- 测试：Vitest + jsdom

**数据库与基础设施：**
- PostgreSQL 15，采用多租户隔离架构
- Docker 多阶段构建 + Docker Compose 编排
- Nginx 反向代理

### 核心架构亮点：多租户数据隔离

这是本项目在架构设计上的一个重点。系统采用 **Database-per-Tenant** 模式：

1. 主数据库（main_db）仅存储用户凭证和租户映射关系
2. 每个用户注册时，系统自动从 `tenant_template` 模板克隆出一个独立的 PostgreSQL 数据库（`tenant_<username>`）
3. 请求进入后端时，中间件从 JWT 中提取租户信息，将对应的数据库连接附加到请求上下文中

这种设计实现了用户数据的物理隔离，安全性高，且便于后续的数据备份和迁移。

## 四、技术难点与解决方案（约3分钟）

### 难点一：超星自定义字体反爬

超星平台使用自定义字体文件将页面文字映射为不可直接读取的 Unicode 编码，直接抓取到的文本是乱码。

**解决方案：** 使用 fonttools 库解析字体文件中的 cmap 表，建立字符映射关系，将加密文本还原为可读内容。

### 难点二：多平台加密协议兼容

超星和智慧树各自使用不同的加密方案（AES、DES 等）来保护接口参数。

**解决方案：** 通过逆向分析平台前端 JavaScript 代码，还原加密逻辑，使用 pycryptodome 和 pyaes 在 Python 端实现对应的加解密流程。

### 难点三：多租户连接池管理

每个用户对应一个独立数据库，如果为每个请求都新建连接，性能开销巨大。

**解决方案：** 使用 psycopg2 的 ThreadedConnectionPool 实现连接池，对主数据库和各租户数据库分别维护连接池，支持懒加载初始化，避免启动时一次性创建所有连接。

### 难点四：后台长任务管理

课程学习是一个长时间运行的任务（可能持续数小时），不能阻塞 HTTP 请求。

**解决方案：** 设计了内存任务存储（TaskStore），任务提交后立即返回 task_id，前端通过轮询 `/api/v1/course/status/{task_id}` 获取实时进度。
